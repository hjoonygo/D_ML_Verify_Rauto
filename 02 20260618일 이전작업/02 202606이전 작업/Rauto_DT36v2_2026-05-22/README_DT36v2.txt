[왜 다시 도나 — 옛 DT36와 뭐가 다른가]
 옛 DT36는 '구멍 안 고친' 엔진이라 SHORT 5건이 역방향 90일 폭주(-$140k)로 망가졌다.
 통합판_v2 문서의 수정("진입 즉시 하드플로어 항상 ON")을 엔진에 적용(Exec_GridD_v2)하고,
 36개월 하락장을 손절폭 4종 × 비율 2종으로 연도별 재검증한다.

[수정 내용 (문서 명시)]
 진입가 기준 하드플로어(ROE %) 손절을 '모든 단계에서 상시' 검사 -> 첫 OB 도달 전에도 손절 작동.
 grid 손절폭: 2.4 / 3.0 / 5.0 / 8.0 %(ROE).  비율: 50:50 / 60:40.  혁신1 ON, trigger 15, ext 0.65.
 * 버그 교정이라 2.864 '정확 재현'은 더이상 목표 아님(고친 만큼 청산이 달라짐).

[실행법]
 1) 이 zip을 하위 폴더에 푼다. 예) D:\ML\Verify\Rauto_DT36v2_2026-05-22\
    (Merged_Data_with_Regime_Features.csv 가 상위 D:\ML\Verify 에, feat_struct_8 컬럼 필요)
 2) python run_downtrend36_v2.py     (pandas, numpy). 약 1.5~2.5분.

[화면에서 볼 것 — 판정 순서]
 (1) 각 config '폭주(90일강제) __건' -> 0 이면 구멍 수정 성공.
     * 데이터 끝 경계 거래 1건은 자동 제외함(자료부족 아티팩트, 진짜 폭주 아님).
 (2) 요약표 '전연도PF>1=True' 인 손절폭 -> 그게 진짜 하락장 챔피언(어느 해든 통함).
 (3) 그 중 MDD 작고 순익 큰 손절폭이 최적. 50:50 vs 60:40 비교.

[결과물 - 회신]
 콘솔 캡처(요약표 + 연도별 줄) + DT36v2_summary.csv + DT36v2_*.csv 를 zip 1개로.
