[0. 시스템 개요 — 검토 맥락]
- BTC 선물 자동매매. 36개월 백테 검증된 추세봇(챔피언) + 변종 7봇(R1~R7). 현재 전부 페이퍼(가상).
- 인프라: AWS Windows 인스턴스(7/1 도쿄 ap-northeast-1 이전 예정, 매칭엔진 동일리전 저지연).
  Python control_server(표준라이브러리만, 포트 8787), 폰 PWA 대시보드, Tailscale 사설망(serve, funnel 미사용).
- 알림: Telegram 봇(env 토큰). 성과리포트: Gmail(앱비번 env) + Telegram 동시 발송.
- 7/1 소액 실거래 예정 — ★현재 '실주문 실행코드'는 없음(전부 페이퍼), 라이브 주문모듈 신규구축 필요.
- 지인 몇 명에게 '수익확인용' 읽기전용 배포 예정(타인 자금 운용 아님, 성과 열람만 — 1단계).
- 비밀(거래소키·Gmail앱비번·텔레그램토큰)=OS 환경변수(setx)로만, 코드/깃/채팅 미포함 원칙.
- 규모: 1인 운영, 소액, 지인 소수. ★엔터프라이즈급 과잉설계 말고 '소규모 실용 최적' 관점으로 검토 바람.

요청: 아래 12개 분야 각각에 대해 ① 빠진 것 ② 취약점 ③ 우선순위(필수/권장/선택) ④ 구체적 권고를 주세요.
      특히 '지인 배포'와 '소액 실거래' 두 전환점에서 반드시 막아야 할 치명적 구멍을 짚어주세요.

[1. 비밀 관리 (Secrets)]
현재/계획: 거래소키·Gmail앱비번·텔레그램토큰 모두 env(setx /M)로만. 코드·git·로그·채팅 미포함 원칙.
질문: env 단독으로 충분한가? (Windows 레지스트리 env 평문 위험?) 비밀 회전 주기·방법? 로그/트랜스크립트 누출 점검법?
      비밀관리자(vault) 도입이 소규모에 과한가, 필요한가?

[2. 거래소(Binance) API 키]
현재/계획: 7/1 실거래 키 = 선물 활성 + 읽기+거래만, 출금 절대 비활성, IP화이트리스트(Elastic IP), 2FA, 90일 회전.
질문: IP화이트리스트만으로 키 탈취 위험 충분히 막히나? 거래전용 키도 남용 가능한가(공격자가 의도적 손실 유발)?
      서브계정 분리·주문 한도 설정이 필요한가? recvWindow·시계동기 보안 측면 이슈?

[3. 네트워크 / 접근 제어]
현재/계획: Tailscale 사설망(serve, HTTPS). funnel(공개) 미사용. 8787 공인망 직노출 안 함.
질문: Tailscale 계정 자체가 뚫리면? (Tailscale 2FA·키 만료 권장설정?) ACL로 기기·포트 제한 필요?
      AWS 보안그룹(방화벽) 인바운드 최소화 점검 포인트?

[4. 제어 엔드포인트 인증]
현재 약점: control_server의 /cmd(강제청산·정지·flat)·/email 이 토큰 없이 tailnet 내 누구나 호출 가능.
계획: 관리자 토큰(env) 있을 때만 제어 허용. 읽기(/state.json)는 분리.
질문: 토큰 방식(헤더 Bearer? HMAC 서명?) 권고? 토큰 유출 대비(만료·회전)? CSRF·재생공격 우려? 감사로그 필요?

[5. 지인 배포 (읽기전용 분리)]
계획: 뷰어=읽기전용(/state.json·대시보드 제어버튼 숨김), 제어·이메일은 캡틴 토큰만. 사람별 개별 접근(공유비번 금지).
질문: '제어버튼 숨김'만으론 불충분(서버단 거부 필수)? 사람별 접근을 Tailscale 기기초대 vs 토큰링크 중 무엇이 안전·관리편한가?
      화면캡처/재배포로 알파·계좌정보 유출 대비(워터마크·계좌정보 마스킹)? 접근 폐기(revoke) 절차?

[6. 코드 / 알파 보호 (난독화)]
현재: PyArmor 난독화 작동 증명(알파 수치 0노출·실행정상). 단일파일(.exe) 패키징은 미완. trial 버전.
검토결론(우리): 어떤 난독화도 100%아님(메모리덤프·정적분석) → 진짜안전=핵심 알파 서버보관+배포본 미포함.
질문: 지인배포엔 Nuitka(무료) vs PyArmor(유료) 어느 쪽 권고? '알파 서버측 보관 + 클라이언트는 연결/뷰어만' 아키텍처가 현실적인가?
      난독화 우회 외에 코드유출 경로(메모리·디스크캐시·.pyc)도 막아야 하나?

[7. 서버 가용성 / 하드닝 (단일장애점)]
현재 약점: control_server가 수동 cmd창에서 실행 → 창 닫히면 다운(SPOF). 스케줄작업 자동재시작은 SYSTEM 계정 python경로 문제로 실패한 이력.
계획: python 전체경로로 스케줄작업(부팅시 자동·재시작) 재등록. Telegram 하트비트 모니터. 파일권한·유저격리.
질문: Windows 서비스(nssm 등) vs 스케줄작업 중 권고? 다운 감지·자동복구 패턴? 무중단 위해 꼭 해야할 최소 하드닝 목록?
      OS·python 패치 주기? 단일 인스턴스 = SPOF인데 소액단계서 이중화는 과한가?

[8. 공급망 (의존성)]
현재: control_server=표준라이브러리만(공급망 위험 낮음). 봇 엔진=pandas/numpy/smartmoneyconcepts 등.
질문: 의존성 핀고정(requirements 잠금)·해시검증 필요? smartmoneyconcepts 같은 소형 패키지 신뢰성 점검법? 정기 취약점 스캔(pip-audit) 권고?

[9. 라이브 거래 안전 (실거래 코드)]
현재: 실주문 코드 없음(신규구축). 계획=서명주문+체결확인+계좌조회 + 코드 하드캡(최대명목·최대레버·분당최대주문·사이즈 sanity).
      테스트넷 생략, 대신 실거래 1회 스모크주문으로 검증(캡틴 결정).
질문: '코드 하드캡'에 반드시 넣어야 할 안전장치 목록? 오발주(사이즈10배·방향반대·무한루프) 방지 패턴? 레이트리밋 백오프·중복주문 방지(idempotency)?
      테스트넷 생략 + 1회 스모크 방식의 위험도 평가 — 더 싸고 안전한 대안 있나? 킬스위치(긴급청산) 신뢰성 확보법?

[10. 로깅 / 데이터]
현재: 비밀은 로그 미기록 원칙(이번에 echo로 1회 노출된 교훈 → 값 출력 금지 규칙화).
질문: 거래로그·상태파일에 민감정보(잔고·포지션) 보호? 로그 보존·접근통제? 채팅/AI 트랜스크립트에 비밀 안 남기는 운영규칙?

[11. 법 / 커스터디 (타인 자금)]
현재: 1단계는 '수익 열람만'(타인 자금 미수령). 만약 지인 자금 운용 시 = 한국 자본시장법 무허가 투자일임 리스크.
질문: '성과 열람 공유'와 '자금 운용 위임'의 법적 경계? 자금 안 받고 성과만 보여주는 것도 광고·권유로 간주될 위험? (전문가 상담 전 일반적 주의점)

[12. 백업 / 복구]
현재: 코드 git(GitHub origin). 데이터 D드라이브+G드라이브. 비밀은 env(백업 안 됨).
질문: 인스턴스 장애 시 복구 절차(코드·설정·키 재설정) 런북 필요? Elastic IP·키 분실 대비? 상태파일(state.json)·원장 백업 주기?
