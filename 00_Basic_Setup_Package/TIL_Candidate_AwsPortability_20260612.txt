[한 줄] 운영지(AWS) 이식 시 첫 관문은 절대경로와 pip 의존성이다.
        오염검사는 '원본 파일 대조'가 아니라 '확정 해시 상수 대조'여야 이식 가능하다.

[사건] PC에서 13/13 PASS였던 Stg14가 AWS에서 즉사 2건:
  ① smartmoneyconcepts 미설치(ModuleNotFoundError) — PC엔 깔려 있어 못 느꼈던 의존성.
  ② check.py가 D:\ML\verify의 원본 파일과 대조 — AWS엔 그 경로 자체가 없음.

[교훈 3개]
1) 오염검사의 본질은 '이 파일이 검증된 그 바이트인가' — 기준은 원본 '파일'이 아니라
   확정 '해시값'이다. 해시 상수를 코드에 박으면 원본이 없는 운영지에서도 검증된다.
   (PC에서는 해시상수+원본파일 이중 검증, 운영지에서는 해시상수만 — v4 패턴)
2) pip 의존성은 import 전수 스캔으로 기계적으로 추출해 run.bat에 설치줄로 내장 +
   README 점검표에 목록 박제. "내 PC에서 돌았다"는 의존성 증명이 아니다.
3) 경로 원칙: 데이터 = 공통 절대경로(C:\BinanceData), 작업 = %~dp0/스크립트 상대경로만.
   기록(INDEX)은 본진 경로 부재 시 로컬 로그(aws_workhstr.log)로 자동 전환.
   원본 보존 파일(§8) 내부의 D:\ 탐색 경로는 수정 금지 — 미사용 경로면 무해함을 확인만.
